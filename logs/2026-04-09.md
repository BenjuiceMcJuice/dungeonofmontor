## Iteration 1 — scene/montor split (voice tense)

User flagged that the demo has TWO voices: italic third-person scene description (atmospheric DM narration) and MONTOR: dialogue (first-person Montor speaking). The PoC was collapsing both into one labelled "MONTOR" block, which killed the contrast and the mask-slip mechanic.

**Fix:** updated JSON contracts to return `{ scene, montor, hook, ... }` separately. Scene = atmospheric DM-mode, italic Goudy, no label. Montor = first-person speech, labelled MONTOR in pixel-purple, only when he actually speaks. Backward compat preserved for legacy `content`-only messages.

Also changed: the rat-fixation in the theme anchor. The demo's One-Ear-the-rat scene was illustrative, not canonical, so I generalised the bestiary block to list multiple creature types and explicitly told Groq *"the same player should not always meet rats"*.

## Iteration 2 — testing the first version

User played a session as ASHSLUG with Estate Agent Montor. Saved to `docs/testing_feedback/narrative test.txt`.

### What was wrong
- **Montor was a tour guide, not a DM.** Long descriptive monologues, "what do you do next?" (literally violated my own forbidden phrase rule), no events thrown at the player
- **Nothing ever happened.** Player sat by pond → Montor described pond. Player threw stone → ripples spread. The DM never made anything happen TO the player.
- **Failed jump roll consequence was aesthetic** ("the birds stopped singing") not concrete (no HP loss, no item lost)
- **Opening was four solid MONTOR paragraphs** with no SCENE block at all

### Diagnosis
The voice rules told Groq HOW to describe and HOW to talk but didn't tell it that **a DM's job is to make things happen TO the player, not to wait for the player to make things happen TO the world.** The proactivity was missing.

## Iteration 3 — proactivity rules + pacing intelligence

Major architectural addition. Locked Montor to `bad_montor` (the only personality, beefed-up description), built a proper DM rule system, and added app-side pacing intelligence.

### Voice/personality changes
- Personality picker removed from setup screen (PoC v1 lock)
- `bad_montor` description rewritten: theatrical, dangerous, watching, master plan, escalation looks like traps/locked doors/creatures/voice-getting-closer/HP damage. Mask slips into *deeper* villainy (not kindness) when family things are touched.
- All other 16 personalities marked DEPRECATED in code, kept for future re-enable

### New `MONTOR_DM_RULES` block in Tier 1
- **Every turn must include a hook** — a concrete event/change that demands a reaction. Required JSON field.
- **Escalate the thread the player is touching.** Don't invent new threats. Read what they're engaging with, make THAT thing weirder.
- **Passive player → deepen, not break.** Don't match low energy with low energy. Reward attention by making the thing they're attending to stranger.
- **Failed rolls have teeth.** Concrete `hpChange`/`itemGained`/`newThreads`, not aesthetic-only.
- **Forbidden phrases** (hard list): "what do you do", "the choice is yours", "the path stretches before you", "what next", "the question becomes", "as you stand there", "ready to be explored", "isn't it lovely".

### App-side pacing intelligence
New helpers in `narrative.js`:
- **`computePacingState(campaign)`** — walks recent messages to calculate `quietStreak` (consecutive narration turns with no concrete consequence), `pressureTarget` (low/medium/high), `activeThread` (matches player action keywords against open threads), `playerEngagement` (active vs passive heuristic).
- **`buildPacingDirective(pacing)`** — renders a dynamic instruction block injected at the END of every Groq user message (LLMs weight prompts most heavily there).
- Quiet streaks: 0-1 = `low` (let it breathe), 2 = `medium` (deepen), 3+ = `high` (escalate now, use a thread, inflict harm).

Key insight: **the LLM is good at generating, bad at deciding when to escalate.** App does the deciding (deterministic), LLM executes the directive.

### Required `hook` field
A required JSON field forces the model to articulate a concrete event each turn. Backed up by validation rules in the prompt. Renders in the feed as a gold-accent italic block below scene + Montor speech.

### Messages now store consequences
Each narration message records `{ scene, montor, hook, consequence }` so the pacing logic can walk history backwards and detect quiet streaks.

## Iteration 5 — rate limits, then major refactor

User started hitting Groq rate limits. Free tier on `llama-3.3-70b-versatile` is roughly 30 RPM / 12K TPM, and the two-call dice flow with ~2500 input tokens per call put us at maybe 6 dice actions per minute under stress testing.

### First mitigation: retry with backoff
- Added `parseRetryDelay()` to read Groq's "try again in X" hint
- Wrap `callGroq` in single-retry logic with up to 8s wait
- Added `MAX_RETRY_WAIT_MS = 8000` so very long waits fail fast instead of silently hanging
- Added `FETCH_TIMEOUT_MS = 25000` with `AbortController` so hung requests don't lock up the UI
- Added live error display under the busy button so users see what's happening during retry

### Big architectural refactor: single-call dice flow
Replaced the two-call flow (`assessAction` then `narrateOutcome`) with a single `assessAndNarrate()` that returns BOTH success and failure branches in one Groq response. The app rolls the dice locally and picks the matching branch.

**New return shape for dice actions:**
```js
{
  needsRoll: true,
  stat: "agi",
  dc: 12,
  narrateBefore: "atmospheric build-up",
  success: { scene, montor, hook, hpChange, itemGained, ... },
  fail: { scene, montor, hook, hpChange, itemGained, ... }
}
```

**Net effect:**
| Metric | Before | After |
|---|---|---|
| Groq calls per dice action | 2 | 1 |
| Input tokens per dice action | ~5,000 | ~1,800 |
| Total tokens per dice action | ~5,500 | ~2,700 |
| Round-trips (latency) | 2 | 1 |
| Free-tier dice actions/min | ~6 | ~16 |

### System prompt trim
Collapsed `MONTOR_THEME_ANCHOR + MONTOR_VOICE_RULES + MONTOR_DM_RULES + MONTOR_DICE_RULES + MONTOR_SAFETY_RULES + MONTOR_JSON_RULES` into one unified `MONTOR_CORE_RULES` block. Deduplicated. Tightened. Removed `MONTOR_JSON_RULES` entirely (the `response_format: json_object` mode already enforces it).

Token count: ~2300 → ~1100 for Tier 1.

### Deprecated `assessAction` and `narrateOutcome`
Kept as thin stubs (assessAction → delegates to assessAndNarrate, narrateOutcome → returns null) so any imports still compile.

## Iteration 7 — testing 8B + the design verdict

User played a session as FUCKER on 8b with the new proactivity rules. Saved to `docs/testing_feedback/narrative test 8b.txt`.

### What 8B got right
- Structure (scene + Montor + hook + dice) all rendered correctly
- Hooks were concrete events (humming noises, doors opening, hand reaching)
- Failed rolls had narrative consequences (tripping, falling, lurching)
- Mocking Montor voice landed
- No "what do you do?" violations

### What 8B got wrong (and why it's not fun)
- **Repetitive hook templates.** Every hook was some form of *"a faint X begins to build"* — at least 7 times across the session. The 70b model would naturally vary these. 8b can't.
- **Looping mockery.** Montor's lines repeated almost verbatim ("you really should be more careful where you step / what you seek / where you're walking"). The "garden party" thread was teased 6 times and never delivered.
- **Zero HP damage in the entire session.** The DM rules say *"Use hpChange freely (1-4 per hit)"* — 8b ignored that instruction completely. No items lost, no real stakes.
- **Forward momentum was illusory.** Every other turn the AI said *"you're getting closer"* / *"the heart of the garden"* — but nothing actually happened. 12+ actions in, the player asked *"More Bees again?"* — recognising the pattern.
- **No mask slips.** The whole bad_montor "mask slips when you touch family things" instruction requires the model to recognise emotional beats and connect them to lore. 8b can't.

### Diagnosis
**8b is too weak to execute the proactivity rules with depth.** It hit the *form* (scene/montor/hook structure, mocking voice, dice flow). It missed the *substance* (variation, thread tracking, real consequences, mask slips, progression). This is a model limit, not a prompt limit. The 70b model would already be doing all of this from the same prompts.

**Two important things this proved:**
1. The architecture is sound. PoC machinery works. The first-pass quality issue was prompt design, the rate-limit issue is a free-tier infra constraint, the 8b weakness is model selection — all solvable independently.
2. The 70b model is essential to the experience. The gap between 8b output and a 70b run isn't subtle, it's the entire game. **The gap IS the game.**

## The multiplayer key insight

User suggested: in multiplayer narrative mode, each player could use their OWN Groq/Claude key, with the AI response saved to Firestore and shared. **This is genuinely the right architectural answer**, not just a clever optimisation:

- Each player burns their *own* per-account quota
- 4 players = 4× the effective rate limit for free
- The expensive 70b model becomes viable for parties on free tier
- The shared Firestore message log means everyone reads the same DM regardless of who paid the token cost
- Solo play stays cheap (paid Groq/Claude is a few dollars for hundreds of sessions)

This should be codified in the narrative_mode_spec.md as the official Phase 2 multiplayer model — not just a feature, but the actual unlock for the whole game economy.

## Lessons learned today (for next session's context)

1. **Voice rules without proactivity rules produce a tour guide, not a DM.** The model defaults to "describe a peaceful place and wait" unless explicitly told to make things happen. This was the biggest single-win fix.

2. **App-side state, LLM-side generation.** Pacing decisions (when to escalate, which thread to touch) belong in deterministic app code. The LLM only executes directives. Asking the LLM to "decide when to escalate" is unreliable.

3. **Required JSON fields shape output more than instructions.** The required `hook` field forced concrete events into every turn — better than any number of "make sure to include a hook" instructions.

4. **Model quality matters more than prompt cleverness for narrative games.** 8b with the best prompts is worse than 70b with mediocre prompts. The "depth" the player feels is mostly model intelligence, not prompt engineering.

5. **The single-call optimisation is strictly better.** Half the requests, half the latency, ~33% fewer total tokens, same output quality. Should never have been two calls in the first place — that was an over-cautious early design choice.

6. **Groq free tier is too tight for testing**, but probably fine for actual play if someone is reading + thinking between actions. Don't conflate stress-testing constraints with real-play constraints.

7. **Rate limits are per-account, not per-key.** Don't try to "isolate" by making new keys/projects on the same account. Different account or different provider is the only real reset.

8. **Multiplayer with shared keys is a stronger architectural answer than I initially gave it credit for.** Worth elevating in the spec from "feature" to "the actual scaling model".
