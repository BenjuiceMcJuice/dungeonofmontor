**Partially defined (examples exist, full schema gaps):**

- `/floors/.../zones/{zoneId}` — one example (Garden). Missing: enemy roster IDs, loot table ref, description pools, door sprite theme, background colour/gradient.
- `/enemies/{enemyId}` — one example (Rat). Stat fields don't match code (spec: atk, code: str/agi/def/int). Spec: spriteId, code: archetypeKey. Name pools in spec 05 have 4-5 names/tier, code has 1/tier.
- `/runs/{runId}/grid/{chamberId}` — one example. Missing: montorState sub-doc.

**Not defined at all:**
- `/items/{itemId}` — no schema anywhere. Build plan lists Stage 1 items but no doc shape.
- `/lootTables/{tableId}` — referenced in multiple specs, never defined.
- `/npcs/{npcId}` — collection listed, zero detail.
- `/events/{eventId}` — collection listed, zero detail. Montor events, narrative events, triggers.
- `/zoneEffects/{effectId}` — listed, never defined. Effect name, mechanical impact, visual indicator.
- `/defaultValues/{valueId}` — example for multiplayer scaling only. Needs full balance value catalogue.
- `/chamberTemplates/{templateId}` — one example as sub-collection. Needs full field list including description pools, trap damage ranges, rest values, merchant refs.
- `/runs/{runId}/montorState` — not specced. Needs: loot mod, difficulty mod, pending offers/curses, locked doors, messages, mood.

### Spec vs code conflicts identified

| Field | Spec (08) | Code (enemies.js) | Resolution |
|---|---|---|---|
| Enemy combat stats | `hp, atk, def, agi` | `hp, str, agi, def, int` | Code is richer — use code version as canonical |
| Enemy names per tier | 4-5 names (spec 05) | 1 name (code) | Expand code to full pools when migrating to Firebase |
| Stat minimum floor | 8 (spec) | 4 (code) | Decide: 4 allows weak-feeling Dust mobs, 8 matches D&D feel |
| Boss design | "zone boss required for key/stairwell" | Single global Void orc | Need one boss per zone |
| Chamber templates | Sub-collection on zone doc | Hardcoded array in dungeon.js | Move to Firestore |

### Architecture plan agreed

The engine (maze gen, combat, dice, navigation) stays in code. Content (floors, zones, enemies, items, loot tables, chamber templates) moves to Firebase. This means adding a floor = adding Firestore docs, not writing code.

**Build order:**
1. Write complete Firestore schemas for all content collections
2. Seed Firebase with Garden data (migrate what's hardcoded)
3. Refactor engine to read from Firestore instead of hardcoded arrays
4. Build `loot.js` + item catalogue in Firebase
5. Build floor transition pattern (safe room + Montor encounter between floors)
6. Seed remaining floors/zones in Firebase (templates, enemy rosters, placeholder descriptions)
7. Inventory UI + consumable usage in combat
8. Boss encounters + run completion
9. Montor state on run doc + scripted between-floor lines
10. Visual polish per floor

### Floor transition design (new concept)

When descending a stairwell between floors:
```
Stairwell Descent → transition screen → Safe Room (rest, can't go back)
→ Montor Encounter (scripted Stage 1, AI Stage 4, human multiplayer)
→ New zone doors appear
```

Montor encounter is the personality moment — taunts, offers, threats. All three Montor modes (AI, human, scripted) write to the same field on the run doc. The game renders whatever's there. Player can't tell who wrote it.

### Montor influence system — three modes, one data shape

All Montor actions are data writes to `/runs/{runId}/montorState`:
- AI Montor (Groq) reads game state → decides → writes
- Human Montor uses Console UI → same writes
- Scripted Montor (Stage 1) → static writes per floor

Game engine checks montorState each chamber entry and applies. The interface is identical regardless of who's pulling the levers.

## Key Decisions
- Content moves to Firebase; engine stays in code
- Data model schemas need writing before any Firebase seeding
- Enemy stat fields: use code version (str/agi/def/int) as canonical, update spec to match
- Floor transition pattern: safe room + Montor encounter between every floor
- Montor modes all write to same data shape — engine doesn't care who's driving
